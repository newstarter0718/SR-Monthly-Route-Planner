系統運算邏輯說明

SR Monthly Route Planner V18.1 · Planning Engine Technical Reference

整體架構概覽

規劃引擎分為六個階段依序執行。每個階段輸入上一階段的結果並加以優化：

1

地理加權 K-Means 分群

將客戶依地理位置與工作量分配到 N 個日度區域（N = 工作天數）

2

空間內聚性驗證 (Spatial Coherence) V17 新增

偵測各區域內的地理離群點，重新指派到更緊湊的鄰近區域

3

工作量平衡優化

迭代執行「移動」與「交換」操作，最小化各工作日的工作量標準差

4

動態週排程 V16 新增

依頻次將客戶分配到四週中的具體日期，考量週負載與地理鄰近性

5

每日路線最近鄰優化 (TSP近似)

對每個工作日執行最近鄰演算法，產生有序的拜訪序列

6

距離與時間計算

以 Haversine 公式計算相鄰門市間距離，估算行駛時間

Stage 1：地理加權 K-Means 分群

目標

將所有客戶分配到 K 個區域（K = 工作天數，5 或 6），每個區域代表業代一天的服務範圍。

工作量定義

每位客戶的「月度工作量」為拜訪頻次乘以四週：

monthly_load(c) = frequency(c) × 4

例如：frequency=4（每週一次）→ monthly_load = 16；frequency=1（每月一次）→ monthly_load = 4

初始化方式

採用 K-Means++ 初始化策略：第一個中心點隨機選取；後續中心點以「與現有中心點距離平方」為機率權重選取，確保初始中心點分散。

迭代收斂

repeat until centroids converge (max 100 iterations): // Assignment step for each customer c: nearest_zone = argmink haversine(c.lat, c.lng, centroid_k) // Update step (workload-weighted centroid) for each zone k: centroid_k.lat = Σ(c.lat × monthly_load(c)) / Σ monthly_load(c) centroid_k.lng = Σ(c.lng × monthly_load(c)) / Σ monthly_load(c)
⚠️ 注意：使用加權重心（而非簡單平均）可確保高頻客戶密集區域的中心點偏向實際工作負荷的重心，而非地理中心。

Stage 2：空間內聚性驗證 (V17)

問題背景

標準 K-Means 雖可平衡工作量，但可能產生「地理上分散」的區域——同一天的路線包含多個分散的群落，造成大量回頭路。

離群點偵測演算法

  1. 計算各區域的「核心聚落」：每個客戶與其所在區域中心點的距離
  2. 計算區域內所有距離的 75 百分位數 (P75) 作為基準閾值
  3. 閾值 = P75 × 1.5
  4. 距離超過閾值的客戶標記為「離群點」
outlier_threshold(zone) = percentile75(distances_in_zone) × 1.5

重新指派邏輯

離群點被重新指派到「地理距離最近且工作量未超載」的鄰近區域，確保：

  • 各區域在地理上保持緊湊
  • 沒有任何區域因接收離群點而超過容量上限

Stage 3：工作量平衡優化

優化目標

最小化各工作日月度工作量的標準差，使每天的拜訪門市數盡可能均等。

objective = minimize σ(monthly_load_per_zone)

操作類型

操作說明評分公式
Move（移動） 將一個客戶從高負載區域移到低負載區域 score = varianceReduction × 0.7 − geographicCost × 0.3
Swap（交換） 交換兩個不同區域的客戶（工作量相近但地理位置互換更合理） score = varianceReduction × 0.7 − geographicCost × 0.3

收斂條件

迭代直到以下任一條件成立：

  • 工作量標準差 σ < 2.0
  • 連續 3 次迭代無改善
  • 達到最大迭代次數上限

Stage 4：動態週排程 (V16)

四週循環結構

所有規劃以 固定四週 為單位，日期格式為 W{week}-{day}（例：W2-Wed）。

Frequency月度總次數排程策略
8（每週兩次）32次主要日 + 輔助日（每週兩天）
4（每週一次）16次分配到各週的主要日
2（每月兩次）8次分配到 W1、W3 的主要日
1（每月一次）4次分配到 W1 的主要日

高頻客戶（Frequency=8）輔助日選擇

對於每週需拜訪兩次的客戶，輔助日依以下加權評分選擇：

aux_day_score(day) =
− (weekly_load[day] × 0.6) // 負荷低的日期優先
− (distance_to_centroid × 0.4) // 地理鄰近的日期優先

Stage 5：每日路線最近鄰優化

演算法說明

採用 最近鄰啟發式演算法（Nearest Neighbor Heuristic），為 TSP（旅行推銷員問題）的近似解：

  1. 以區域中心點作為起點
  2. 從起點出發，每次選擇「尚未拜訪中距離最近」的客戶
  3. 重複直到所有客戶均完成排序
reoptimizeDailyRoute(visits, zone_centroid): route = [] current = zone_centroid remaining = [...visits] while remaining.length > 0: nearest = min(remaining, key=haversine(current, v)) route.push(nearest) current = nearest remaining.remove(nearest) return route (with updated Visit_Sequence)
複雜度：O(n²)，對於每日 15–40 個門市的規模，執行時間在毫秒內完成。

Stage 6：Haversine 距離公式

所有兩點間距離均使用 Haversine 公式計算，考量地球曲率，單位為公里：

haversine(lat1, lng1, lat2, lng2):
Δlat = (lat2 − lat1) × π/180
Δlng = (lng2 − lng1) × π/180
a = sin²(Δlat/2) + cos(lat1) × cos(lat2) × sin²(Δlng/2)
c = 2 × atan2(√a, √(1−a))
distance = 6371 × c // 地球半徑 6371 km

行駛時間估算

Est_Drive_Time_mins = (distance_km / avgSpeedKmph) × 60

容量計算與 Spillover 邏輯

每日需求計算

required_daily_visits = Σ monthly_load(c) / (workDays × 4 weeks)
required_limit = max(daily_load_per_day_across_4_weeks)

Spillover 觸發條件

當某一工作日排入的門市數超過 dailyVisitLimit 時，多餘的門市進入 Spillover 清單，並記錄溢出原因。
